# Agent 编排学习记录

> role：你是一名 agent 开发学习者，通过设计该项目学习 agent 编排

## 1. 项目一句话
这个项目不是“写几个工具给 LLM 调用”，而是学习如何围绕模型搭建一套 **可调用、可控制、可测试、可扩展** 的 Agent 执行系统。

## 2. 主链路
`CLI -> Request Model -> AgentOrchestrator -> analyze -> execute_tools -> format_result -> structured response`

这条链路说明：
- `CLI` 负责收集用户输入
- `Request Model` 负责把输入变成结构化参数
- `AgentOrchestrator` 负责编排
- `analyze` 负责让模型决定“要不要用工具、用哪个工具”
- `execute_tools` 负责真正执行工具
- `format_result` 负责把结果整理成固定输出

## 3. 这次项目最核心的学习点
- Agent 的本质是“**模型 + 工具 + 状态 + 权限 + 编排**”
- 工具系统最重要的不是“能不能跑”，而是：
- 能不能被模型正确理解
- 能不能被安全地执行
- 能不能被稳定测试
- 能不能在 CI 和跨平台环境里一致运行

## 4. 关键名词解释

| 名词 | 含义 | 在本项目里的作用 |
|------|------|------------------|
| Agent | 能根据目标自主做步骤决策的系统 | 这里的 Agent 会决定是否调用工具、何时停止 |
| 编排 / Orchestration | 把多个步骤按规则组织起来 | `AgentOrchestrator` 负责 5 阶段流程 |
| Tool Calling | 模型按 schema 调用外部工具 | 模型产出 `tool_calls`，编排层解析并执行 |
| Tool | 给模型调用的能力单元 | 例如 `read_file`、`grep_files`、`run_command` |
| Registry | 注册表 | 统一管理当前会话可用工具 |
| Context | 上下文 | 记录当前目标、约束、决策、错误 |
| Schema | 结构定义 | 用来约束输入输出格式 |
| JSON Schema | 一种描述 JSON 数据结构的规范 | 用来告诉模型“参数长什么样” |
| Pydantic | Python 数据模型/校验库 | 用来定义输入模型、生成 schema、做运行时校验 |
| Serializable | 可序列化 | 工具结果必须能转成 JSON/日志 |
| Sandbox | 沙箱 | 受限执行环境，控制命令运行范围和超时 |
| Permission Mode | 权限模式 | 控制工具能不能执行 |
| CI | Continuous Integration，持续集成 | GitHub Actions 这类自动检查环境 |
| MVP | Minimum Viable Product，最小可行产品 | 当前先做最小可用 readonly 工具体系 |
| Fallback | 降级方案 | 模型失败时返回占位结果而不是整个崩掉 |

## 5. Tool 系统的核心抽象

| 抽象 | 含义 | 价值 |
|------|------|------|
| `ToolSafety` | 工具风险等级 | 决定是否可并发、是否需要确认 |
| `ToolSpec` | 工具说明书 | 给模型看：名字、描述、参数、风险级别 |
| `BaseTool` | 所有工具的统一父接口 | 保证工具实现风格一致 |
| `ToolResult` | 工具统一返回信封 | 上层不用猜每个工具怎么返回 |
| `ToolRegistry` | 工具发现入口 | 编排层只从这里拿工具 |

## 6. `ToolSafety` 为什么重要
`ToolSafety` 分三类：
- `readonly`：读操作，无副作用
- `write`：会改文件或仓库
- `execute`：会执行命令

意义：
- `readonly` 通常可并发
- `write` 和 `execute` 风险更高，需要串行或确认
- 这是一种 **风险分级设计**，不是让所有工具平权

## 7. 什么是“默认最小权限”
“默认最小权限” = **系统默认只暴露最安全、最必要的能力**。

在本项目中：
- 默认 registry 只注册 4 个 readonly MVP 工具：
- `read_file`
- `glob_files`
- `grep_files`
- `list_dir`
- `run_command` 虽然已经实现，但**不默认放入 registry**

这表示：
- 系统默认只给模型“最小必要能力”
- 高风险能力不是不能做，而是**默认不开放**
- 这符合安全设计里的 **Principle of Least Privilege（最小权限原则）**

## 8. 什么是 JSON Schema
JSON Schema 不是数据本身，而是“**数据格式说明书**”。

例如一个工具需要：
- `file_path: string`
- `limit: int`
- `offset: int`

那 JSON Schema 会告诉模型：
- 哪些字段存在
- 字段是什么类型
- 哪些字段必填
- 数值范围是什么

作用有两个：
- 给模型看：帮助模型正确构造参数
- 给程序看：执行前可以校验参数是否合法

## 9. 为什么用 Pydantic
Pydantic 在这里做了三件事：
- 定义输入模型
- 自动校验输入
- 自动生成 JSON Schema

所以它是“**运行时校验 + schema 生成**”二合一工具。

## 10. `description` 为什么很重要
工具的 `description` 不是写给人看的，而是写给模型看的。

它的作用：
- 告诉模型这个工具适合什么场景
- 告诉模型这个工具不适合什么场景
- 告诉模型它有什么约束

所以工具描述本质上是一段“面向模型的使用说明”。

## 11. 为什么所有工具都统一实现 `spec()` 和 `execute()`
这是典型的 **统一接口设计**：
- `spec()`：暴露给模型的说明
- `execute()`：真正执行逻辑

好处：
- 工具之间实现风格统一
- 编排层不用关心具体工具细节
- 后续加新工具只需要实现同样接口

## 12. 什么是 Registry
Registry 是工具注册表，相当于“当前会话有哪些能力”的总表。

它解决的问题：
- 工具如何被发现
- 工具如何被查找
- 工具列表如何统一暴露给模型

没有 Registry，编排层就会到处手写 `if/else` 查工具，系统会迅速失控。

## 13. 什么是 pseudo-tool（伪工具）
本项目里的 `submit_review` 和 `submit_debug` 不是真正访问外部资源的工具，而是“**约束模型输出结构**”的伪工具。

作用：
- 不做 I/O
- 只负责让模型把最终结果按固定字段交回来

这说明：
- Tool Calling 不只是“调用外部能力”
- 也可以用于“约束模型输出格式”

## 14. 什么是 ContextState
`ContextState` 是一次运行中的共享状态，记录：
- 当前目标 `goal`
- 当前约束 `constraints`
- 决策历史 `decisions`
- 当前关注文件 `current_files`
- 错误集合 `errors`

它的意义：
- 让系统不是“每一步都失忆”
- 让后续阶段能读到前面阶段的结果
- 让结果可以复盘和审计

## 15. 什么是 DecisionStep
`DecisionStep` 表示一次关键决策，记录：
- 来自哪个阶段 `phase`
- 做了什么 `action`
- 结果如何 `result`

它的价值是：
- 形成“运行轨迹”
- 方便调试
- 方便解释 Agent 做了什么

## 16. 什么是 ErrorDetail
`ErrorDetail` 是结构化错误，不是简单字符串。

包含：
- 哪个文件
- 哪一类错误
- 错误信息

意义：
- 错误可分类
- 错误可追踪
- 错误可被上层统一处理

## 17. `cwd` 和 `repo_path` 的区别
- `cwd`：当前进程工作目录
- `repo_path`：这次任务真正针对的项目根目录

踩坑点：
- 这两个不一定相同
- 如果路径权限只看 `cwd`，就可能误拒合法文件，或错误放行

所以后来改成：
- 编排层根据 `repo_path` 注入真正的工作区根
- 工具层统一按这个根判断路径合法性

## 18. `ContextVar` 为什么会出现
`ContextVar` 是 Python 用来保存“当前执行上下文中的局部全局变量”的工具。

这里用它做：
- 当前工具调用的工作区根绑定
- 让 `ensure_path_allowed()` 不需要手动层层传参

你可以把它理解为：
- “线程/协程安全的上下文变量”
- 比直接写全局变量更安全

## 19. 为什么要做保守并发
不是所有工具都应该无脑并发。

本项目的策略：
- 只有连续的、并发安全的 readonly 工具才批量并发
- `write / execute` 一律不参与并发批处理
- 结果顺序仍要和原始 `tool_calls` 保持一致

这叫“保守并发”：
- 性能有提升
- 调试成本不会爆炸
- 回放结果更稳定

## 20. 什么是 `is_concurrency_safe`
它表示：
- 这个工具“是否适合和其他工具并发执行”

默认规则：
- `readonly` 认为是并发安全
- 但如果某个 readonly 工具虽然不写文件、却依赖共享状态，也可以覆盖为不安全

## 21. `ToolResult` 为什么必须统一
如果每个工具返回值格式都不同，编排层就会很痛苦。

统一成：
- `ok`
- `data`
- `error`

好处：
- 编排层统一处理成功/失败
- 日志结构统一
- 测试断言简单

这就是“统一返回信封”的价值。

## 22. execute 工具为什么要下沉到 sandbox
`run_command` 本身不应该直接承担所有系统命令执行细节。

所以拆成两层：
- 工具层：参数校验、路径校验、错误语义
- sandbox 层：真正执行命令、控制 timeout、采集 stdout/stderr

这是典型的 **职责分离**。

## 23. 什么是 sandbox
sandbox = 受限执行环境。

在本项目里它至少负责：
- 限制工作目录
- 限制执行时间
- 捕获标准输出和错误输出
- 结构化返回执行结果

意义：
- 防止命令无限挂起
- 防止在错误目录执行
- 防止工具逻辑和系统调用耦合太深

## 24. 什么是 permission mode
当前做了两种：
- `default`
- `plan`

含义：
- `default`：正常模式，工具可按规则执行
- `plan`：只允许计划，不执行工具

这是“系统级权限开关”，而不是单个工具自己的开关。

## 25. 什么是 plan mode
`plan mode` 是一种安全模式。

在这个模式里：
- 允许模型思考
- 允许模型产出计划
- 不允许执行任何工具

价值：
- 当任务风险较高时，先看计划再执行
- 把“思考”和“行动”分开

## 26. 为什么 CI 会影响 execute
CI 环境里通常会有：
- `CI=true`

我们把它当成“非交互环境”信号：
- 高风险工具默认拒绝
- 因为 CI 里没人弹确认框

这体现的是：
- 权限判断不只看工具，还要看运行环境

## 27. 什么是跨平台问题
本地 Windows 能跑，不等于 Linux CI 能跑。

典型问题：
- Windows 专有路径写法
- 路径分隔符差异
- 临时目录权限不同
- 命令行为差异

所以测试不能写死平台假设。

## 28. 什么是 `truncated`
`truncated` 表示“结果是否被截断，不完整”。

注意：
- `返回条数 == limit` 不一定意味着被截断
- 只有“后面还有没返回的结果”才算真的截断

这个是典型边界条件知识点。

## 29. 什么是 fallback plan
当模型失败、客户端不可用、或分析阶段异常时，系统不能直接崩。

所以会返回：
- 占位 review/debug 结果
- 保持结构化输出不变

这叫 graceful degradation（优雅降级）。

## 30. 测试里学到的工程知识
- `DummyTool`：用假工具测试流程，不依赖真实 I/O
- `monkeypatch`：临时修改环境变量/函数/路径，用于隔离测试
- `tmp_path`：pytest 提供的临时目录
- CI-safe test：测试要考虑 Linux/Windows 差异
- 环境变量测试：例如 `CI=true` 和 `PERMISSION_MODE`

## 31. 这次踩过的坑
- 把路径权限错误绑定到 `cwd`
- `truncated` 语义写错
- `CI=true` 导致允许执行测试失败
- 测试里写死 Windows 路径导致 Linux CI 失败
- `Literal + env string` 导致 `mypy` 类型报错
- 本地通过不代表 CI 通过，说明“跨环境一致性”是工程重点

## 32. 这次项目真正让我理解的东西
- Agent 不只是“调模型”
- Tool 不只是“函数”
- 编排不只是“按顺序调用”
- 权限不只是“能不能执行”
- 测试不只是“本地能跑”

更准确地说：
- Agent = 模型决策系统
- Tool = 模型可调用能力
- Orchestrator = 规则与流程控制器
- Context = 运行中的共享记忆
- Permission = 风险控制系统
- Sandbox = 高风险能力隔离层

## 33. 一句话总结
这个项目让我学到：**一个真正可用的 Agent 系统，核心不是“让模型更聪明”，而是把工具、上下文、权限、结构化输出和测试体系组织好。**
